# Opener — Brief technique complet (état au 2026-06-09)

## 1. Vue d'ensemble du projet

**Nom** : Opener (*Open Partitioning Embedding for Named Entity Recognition*).
**Repo** : `d:\Loisir\Code_python\LyRIDS_Opener` — branche `main`, dépôt git propre.
**Cible de soumission** : **LyRIDS Symposium 2026** — IEEEtran double-blind, sources LaTeX dans `paper/`.
**Différenciateur vs OWNER (Alteca, 2024)** : OWNER apprend un encodeur d'entités via Triplet Loss + K-means non supervisé. Opener part d'un embedding pré-entraîné (Nomic v1.5 Matryoshka) + fine-tune contrastif léger + **classifieur supervisé balanced** (SVM/LogReg) par dataset, avec init sur anchor words.

**Architecture choisie (V2 — après contrastive learning)** :

```
Texte
  │
  ▼
MentionDetector ──── GLiNER L (zero-shot, boîte noire)
  │                  [⚠️ NON appelé dans l'éval V2 actuelle — voir note protocole]
  ▼  spans (start, end, text)
Embedder ──────────  Nomic Matryoshka fine-tuné en triplet loss (margin=1, batch=16)
  │                  → outputs/models/embedder_contrastive/
  ▼  embedding (768 dims, truncate_dim=None à l'éval)
Classifier ────────  LinearSVC class_weight='balanced' (fit supervisé par dataset)
  │                  fallback : GMM par label avec init anchor words (V1)
  ▼  label_id
Sortie
```

**Note 1 (V1/V2)** : la V1 (GMM par label + anchor words) reste dans le code mais n'est plus la voie principale pour les expériences quantitatives. Le SVM-balanced post-contrastive a sur-performé.

**Note 2 (⚠️ protocole d'évaluation V2 — point critique)** : dans `scripts/run_balanced_classifiers.py`, l'embedder est appliqué directement sur les **spans GOLD** du corpus ; le `MentionDetector`/GLiNER **n'est jamais appelé**. Les chiffres Opener V2 mesurent donc l'**entity typing sur mentions parfaites** (zéro FN/FP). À l'inverse, les baselines (GLiNER, GNER) sont **end-to-end** avec sentinels FN/FP. → La comparaison n'est PAS apples-to-apples telle quelle. À régler avant le papier : ajouter une variante « Opener end-to-end » (GLiNER MD + SVM, alignement par offsets + sentinels) OU expliciter clairement que la Table 1 oppose *typing-on-gold* à *end-to-end*.

**Note 3 (statut)** : Opener V2 (SVM-balanced contrastif) est **déjà mesuré sur les 13 datasets** — 10 dans `opener_full_bench_contrastive/` (5 juin) + 3 dans `opener_extra_3_datasets/` (6 juin). Il ne reste qu'à les agréger.

## 3. Structure des données

**Format unifié in-memory** : `corpus = list[tuple[text: str, spans: list[tuple[start_char: int, end_char: int, label: str]]]]`. Tous les loaders (`src/data/`) retournent ce format après conversion depuis BIO / CoNLL-U / HF datasets.

**Loaders** :

- `src/data/owner_datasets.py` — registre central des 13 datasets, dispatcher `load_owner_dataset(name, split, max_sentences)`.

- `src/data/gum_loader.py` — clone github.com/amir-zeldes/gum (CC BY-SA), parse CoNLL-U avec extraction `Entity=(ID-TYPE-...)` ; GENTLE fallback sur GUM-train (mêmes labels).
- `src/data/conll_loader.py` — pour CoNLL-2003.
- `src/data/schema.py` — dataclasses `Document`, `Entity`, `MiniDocument` (format OWNER pour interop).

### Phase 1 — Architecture V1 (GMM + anchor words)

   - Init des moyennes via anchor words (`means_init[0]` = moyenne des embeddings des anchor words).
   - Covariance `full` par défaut.
   - Helper `_safe_sample()` pour contourner sklearn issue #15364 (normalisation des poids avant `multinomial`).
   - Threshold OOD basé sur `max(log_likelihood)`.
8. **Anchor words — état réel du code (⚠️ correction)** : le code n'utilise actuellement **que l'anchor BRUT**. [`embedder.embed_anchor_words()`](src/models/embedder.py) encode `task_prefix + mot` sans aucun contexte (« Pas de contexte ici — l'anchor word est encodé seul »), et c'est ce qu'appelle [`pipeline.init_anchors()`](src/pipeline.py). **Aucune** fonction d'anchor contextualisé n'existe dans le code.
   - **Anchor brut** (implémenté) : `embed("person")` — vecteur potentiellement éloigné du nuage des entités réelles ("Marie Curie", "Einstein") embeddées dans leur contexte.
   - **Anchor contextualisé** (à implémenter) : encoder un template type `"[ENT] person [/ENT] is a named entity"` pour rapprocher l'anchor du centroïde des entités réelles.
   - **À FAIRE (ablation demandée)** : implémenter le mode contextualisé dans `embed_anchor_words(contextualized=True)`, puis **comparer brut vs contextualisé** sur la branche V1 (GMM) — l'anchor n'intervient PAS dans la V2 SVM. Résultat à mesurer, pas encore établi (l'ancien Brief affirmait à tort un gain ∼+0.05 et une « rétention » du contextualisé). Voir §10.5 (Table 4) et §10.6 (Fig.).

### Phase 2.5 — Sweep des composants (MD × Embedder × ET)

Objectif : trouver la **meilleure configuration de baseline** (avant contrastive) en faisant varier chaque brique séparément, puis combiner les gagnants.

14. **Sweep Mention Detector (MD)** sur 5 datasets pilotes :
    - GLiNER S, GLiNER M, GLiNER L.
    - Critère : recall des mentions (avant typing).
    - **Gagnant** : GLiNER L (recall ∼0.78 vs 0.72 pour M, 0.61 pour S).
15. **Sweep Embedder** sur les embeddings d'entités contextualisées :
    - Nomic v1, **Nomic v1.5** (Matryoshka), mxbai-embed-large-v1, e5-base-v2, bge-base-en-v1.5, all-mpnet-base-v2.
    - Critère : AMI avec GMM(anchor) fixé en aval.
    - **Gagnant** : Nomic v1.5 — meilleur AMI moyen + Matryoshka native (truncate sans réentraînement).
16. **Sweep Entity Typing (ET)** sur Nomic v1.5 fixé :
    - GMM-anchor, LogReg, LogReg-balanced, LinearSVC, LinearSVC-balanced, KNN.
    - **Gagnant** : LinearSVC-balanced (∼+0.10 AMI vs GMM-anchor, surtout sur datasets déséquilibrés).
17. **Sweep dim Matryoshka** : 64 / 128 / 256 / 512 / 768.
    - **Constat** : 768 légèrement meilleur, mais 256 est un excellent compromis (perte <0.02 AMI, ÷3 mémoire + vitesse).
18. **Config baseline pré-contrastive retenue** : `GLiNER L` + `Nomic v1.5 (768)` + `LinearSVC-balanced` → **AMI ∼0.32** moyen sur les 5 datasets pilotes, déjà au niveau OWNER officiel.
19. **Décision** : à partir de cette baseline, ajouter du contrastive learning pour pousser plus loin (le gap restant vient surtout de classes proches mal séparées par Nomic vanilla).

Macro-F1 moyen e2e (svm-bal) **0.2659** vs GLiNER **0.2823**.

🔴 **VERDICT (à intégrer dans le papier — ne pas l'enterrer)** :
1. **Le « 0.5404 » était un artefact des spans gold.** En pipeline réelle, Opener tombe à **0.3660 AMI**, soit **sous** GLiNER L (0.3888). Opener ne bat GLiNER que sur **3/13** datasets (fabner, gentle, et conll2003 qui est in-domain → leakage).
2. **Opener e2e est dominé** : il utilise GLiNER L comme MD, donc coûte **plus** que GLiNER seul (détection + embedding + SVM), pour un résultat **moins bon**. Tel quel, « GLiNER L tout seul » est Pareto-supérieur.
3. **Le goulot est la DÉTECTION** : recall moyen 0.529, catastrophique sur les verticaux (fabner 0.094, mit_movie 0.141, gum 0.203, gentle 0.24). Les AMI sont dominés par les sentinels FN. Tout gain de typing est plafonné par ce recall.
4. **Signal positif** : le re-typing Opener aide là où le typing natif de GLiNER est le plus faible (fabner, gentle, conll +0.097). Mais insuffisant pour renverser le classement global.

→ **Conséquence** : la contribution du papier ne peut PAS être « Opener bat GLiNER en open-world NER ». Il faut repositionner (cf. §10 / discussion ci-dessous avec l'utilisateur).

## 14. Points importants à ne pas oublier

Ces points sont ressortis dans la conversation mais méritent d'être rappelés explicitement.

### 13.1 Méthodologie expérimentale

- **Une seule seed (42)** sur tous les runs : suffisant pour publication conférence, mais mentionner la limitation.
- **`max_test=1000`** pour homogénéiser le coût — tous les baselines doivent utiliser la même valeur. Documenter dans le paper.
- **AMI traite les sentinels FN/FP** comme labels normaux (cf. Eq. 7 OWNER). C'est pénalisant pour les modèles qui sur-prédisent. **⚠️ Opener V2 (typing-sur-gold) n'a aucun FN/FP** → avantage structurel vs les baselines end-to-end. Ne comparer dans la même table que des protocoles identiques (ajouter la variante Opener end-to-end, cf. §1 Note 2).
- **Fairness mesure énergie** : tous les modèles doivent être mesurés sur la même machine, mêmes conditions thermiques. Faire les runs proche dans le temps (même session).
- **Warmup obligatoire** dans `LatencyMeter` (2 passes) pour exclure la compilation kernel.

### 13.2 Reproducibility

- **Configs YAML par expérience** plutôt que CLI flags multiples — facilite le re-run et le partage.
- **Sauvegarder le JSON de chaque run** avec `params` complets (datetime, hostname, GPU, versions).
- **Pinner les versions** : `requirements.txt` ou `pyproject.toml` à figer avant soumission.
- **Lien dépôt anonyme** (anonymous.4open.science) à préparer pour la review double-blind.

### 13.3 Choix scientifiques à justifier dans le paper

- **Pourquoi 13 datasets et pas 15 ?** → GENIA/i2b2 licence (mentionner explicitement).
- **Pourquoi pas de GENIA proxy ?** → BioNLP-2004 couvre déjà le domaine biomedical.
- **Pourquoi Nomic v1.5 et pas e5/bge/mxbai ?** → Matryoshka native (truncate sans perte), longueur de contexte, license MIT. (À tester en ablation.)
- **Pourquoi SVM-balanced et pas GMM ?** → sur 13 datasets (typing-sur-gold), `linear_svm_balanced` AMI **0.5404** vs `logreg` 0.3728 ; bien au-dessus de la V1 GMM-anchor (~0.20-0.32). À détailler en Sec 5.1.
- **Pourquoi triplet loss margin=1 ?** → cohérence avec OWNER pour comparaison équitable.
- **Pourquoi class_weight='balanced' ?** → datasets déséquilibrés (mit_movie 12 labels avec ratio 1:50), formule sklearn standard.

### 13.4 Pièges de rédaction à éviter

- **Pas d'hallucination de venues bib** — ne jamais inventer le nom d'une conf si on n'a pas le PDF (feedback mémoire).
- **Citer le papier original du dataset** — pas la HF page (feedback mémoire).
- **Pas de "novel" / "naturally" / "we believe"** — style trop marketing.
- **Pas de claims non testés** ("Opener generalizes to any domain") — restreindre aux 13 datasets.
- **Tirets longs `—`** : le user préfère `-` (cf. CLAUDE.md global).

### 13.5 Choses à demander/clarifier au prochain run

- Format des `max_test` : 1000 partout est OK ? Ou prendre tout le test split pour les petits (crossner ~500) ?
- Faut-il un test de significativité statistique (paired t-test sur les 13 datasets) ?
- Faut-il publier un dépôt de code séparé (anonymized) ou attendre la version finale ?

### 13.6 Limitations à expliciter

- **VRAM 6 Go** : T5-xxl et FP16 7B pas testables localement → int4 seulement. Mentionner que les chiffres int4 peuvent différer du FP16.
- **Une seule machine** : pas de mesure variance hardware.
- **Pas de mesure end-to-end** pour OWNER officiel sans cloner le repo.
- **codecarbon TDP fallback** : précision ±20% sur l'absolu, mais rankings préservés (mentionner).
- **Threshold OOD désactivé** dans le benchmark (`-1e9`) pour comparaison équitable — l'OOD est une feature gardée pour les expériences futures.
